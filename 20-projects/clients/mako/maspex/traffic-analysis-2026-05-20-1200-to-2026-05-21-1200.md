## 1. Executive Summary

Pierwsze okno z widocznym efektem zmiany min=20 (poprzednio min=30). Scale-in 30→20 tasków
nastąpił organicznie wieczorem 20.05 (22:27 CEST — AlarmLow memory). Ruch szczytowy wyższy niż W1:

nowych tasków dołączających do puli. Target 5xx spikes 557/509 @ 08:00-09:00 CEST 21.05
korelują z cold-startami podczas scale-out. Anomalny spike CF 376 981 req @ 00:00-01:00 CEST 21.05
wymaga wyjaśnienia (brak korelatu w ALB — prawdopodobnie bot/crawler zaabsorbowany CF).

### Spike Target 5xx @ 08:00–09:00 CEST 21.05

Dwa szczyty 5xx (557 + 509) korelują precyzyjnie z zakresem aktywnego scale-out
20→35 tasków. Nowe taski Fargate potrzebują ~20-30s na warm-up (JVM/Node init, połączenie
z Redis, pobranie konfiguracji). W tym oknie część requestów trafia na jeszcze-nie-gotowe
taski → 5xx. To **expected behavior przy agresywnym scale-out bez pre-warming**.

zarejestruje taski jako healthy przed faktycznym gotowością aplikacji, 5xx są nieuniknione.

| Memory avg | 5.0–8.1% |
| Memory max | 8.7% |

### Mechanizm scale-in 30→20

AlarmLow memory (threshold ~5%) trigger po wieczornym spadku ruchu. Redukcja nastąpiła
stopniowo w ciągu ~2.5h (20:00→22:27 CEST). Redis CurrConnections spadły z ~94 do ~64
— potwierdzenie zamknięcia połączeń przez kończące się taski.

### Mechanizm scale-out rano 21.05

ALBRequestCountPerTarget przekroczył 200 req/task/5min przy 20 taskach i rosnącym ruchu
porannym. Trigger kaskadowy: każdy scale-out (do max=45) obniżał req/task, ale ruch rósł
szybciej niż cooldown pozwalał na stabilizację → 13 zdarzeń.

**Autoscaling policy trigger (obliczenie peak):**
Przy 20 taskach i ~225k req/h = 3 750 req/5min → 187 req/task/5min ← bliski progu 200.
Przy 154 req/s (poranny CF peak) → ALB peak ~89 req/s × 3600 / 12 = ~26 700 req/5min / 20 tasków = ~1 335 req/task/5min ← zdecydowanie > 200 → scale-out uzasadniony.

### Cache hit rate degradacja 89% → 73%

Główna przyczyna: nowe taski dołączające podczas scale-out nie mają lokalnego (in-memory)
cache aplikacyjnego. Pierwsze requesty do każdego nowego taska generują Redis miss (lub
backend DB miss), zanim aplikacja uzupełni swoją lokalną pamięć podręczną.

Przy 15 nowych taskach × ~30s warm-up × ~89 req/s (ALB avg) = ~40 000 requestów
potencjalnie z degradowanym cache. Evictions=0 — Redis jest zdrowy; problem leży
w application-level cache w procesie Node.js/aplikacji, nie w Redis.

## 10. Bottleneck

Bottleneck operacyjny W2: **cold-start cascade podczas scale-out porannego 21.05.**

Łańcuch: min=20 → ruch poranny rośnie → ALBRequestCountPerTarget przekracza 200 → scale-out
→ nowe taski nie gotowe → 5xx + cache miss → graceful degradation.

To nie jest awaria — to feature autoscalingu z nieoptymalnymi parametrami. Identyfikowane
obszary do poprawy:
1. Health check grace period (ALB target group) — czy jest wystarczający?
2. Pre-warming cache / connection pool przy starcie taska
3. Scale-out cooldown — czy zbyt agresywny (13 zdarzeń w ~4h)?

## 12. Recommended next steps

1. **Health check grace period** — sprawdź `deregistration_delay` i `health_check.healthy_threshold`
   dla target group `maspex-prod-api-3000`. Jeśli grace period < 30s, zwiększ do 60s, aby
   uniknąć 5xx podczas scale-out. Terraform: `aws_alb_target_group.healthCheck.timeout` / `target_group.deregistration_delay`.

2. **Scale-out cooldown** — 13 zdarzeń w ~4h wskazuje na zbyt krótki cooldown lub zbyt
   czuły trigger. Rozważ zwiększenie `scale_out_cooldown` z domyślnych 300s do 600s dla
   ALBRequestCountPerTarget policy.

3. **Cache warm-up** — czy aplikacja obsługuje prewarming Redis keys przy starcie? Jeśli nie,
   pierwsze ~30–60s każdego nowego taska generuje excess Redis miss. Opcja: lifecycle hook
   lub startup script wywołujący `GET` kluczowych kluczy cache.

4. **Nocny CF spike 376 981 req/h** — weryfikacja WAF rate limiting dla CF. W1 miał
   podobny wzorzec 4xx (bot traffic). Rozważ WAF managed rule set `AWSManagedRulesBotControlRuleSet`
   dla CF distribution, aby odciążyć origin i zmniejszyć 4xx rate.

5. **Monitorowanie hit rate Redis po scale-out** — dodaj CloudWatch alarm na
   `CacheHitRate < 80%` dla `maspex-prod` jako sygnał cold-cache po scale-out events.
